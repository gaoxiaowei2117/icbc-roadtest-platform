# 后端安全加固清单

PR #5 里随并发修复一起做的几项后端加固。多为"低危/防回归"，但都是廉价且值得的纵深防御。

## 启动即校验关键密钥

`config.py` 里 `jwt_secret` / `worker_api_key` / `secret_public_key` 都默认 `""`，且无启动守卫 → 缺 `.env` 时会用**空 JWT 密钥**静默上线（可伪造任意用户 token）。

> [!info] 线上不可利用，但要防回归
> 实测生产 `.env` 三者均已设置，所以当前不可利用；加守卫是防止将来某次缺 `.env` 的部署翻车。

```python
# main.py lifespan 启动时
def _require_secrets():
    missing = [n for n, v in (("JWT_SECRET", s.jwt_secret),
                              ("WORKER_API_KEY", s.worker_api_key),
                              ("SECRET_PUBLIC_KEY", s.secret_public_key)) if not v]
    if missing:
        raise RuntimeError(f"缺少必需的密钥配置：{', '.join(missing)}（请检查 .env）")
```

> [!note] 为什么放在 lifespan
> 测试用 `TestClient(app)` **不带 `with`**，不会触发 lifespan，所以这个守卫不会干扰测试；生产 uvicorn 正常触发。

## worker key 常数时间比较

`deps.py` 原本 `api_key != settings.worker_api_key`（短路比较，理论时序侧信道）。改用 `hmac.compare_digest`。

## update_profile 越权兜底

`crud/user.py` 的 `update_profile` 只用 `hasattr` 守卫。若将来有人把 `is_admin` 等加进 `UserUpdate` schema，就会变成越权写入点（当前靠 Pydantic `extra='ignore'` 挡着）。加 denylist 作数据层兜底：

```python
_IMMUTABLE_FIELDS = frozenset({
    "id", "email", "password_hash", "is_admin", "is_active", "email_verified",
    "verify_code", "verify_code_expires", "created_at",
})
```

## /auth/refresh 补限流

其它 auth 路由都有 `@limiter.limit`，唯独 `refresh` 没有 → 被盗 refresh token 可无限制重放刷新。补上装饰器 + `request: Request` 参数。

## admin 列表 limit 上界

`admin.py` 的 `limit` 无上限 → `?limit=9999999` 全表查询撑爆 400M 内存的 VPS。改 `Query(default=..., ge=1, le=500)`。

## 经验

> [!tip]
> 这些大多是"现在安全、但契约脆弱"的点。审查时区分**当前可利用** vs **一旦改动就破**很重要 —— 后者写成测试/denylist 钉死，比留注释靠谱。
