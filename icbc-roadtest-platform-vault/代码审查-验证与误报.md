# 代码审查 — 验证方法与误报

本轮用两个并行子代理（安全 + 并发/正确性）扫描后端与 worker，再**逐条对照真实代码与线上系统验证**。核心原则：**审查结论不实测复现，不下定论。** 11 条发现里有 2 条没通过验证。

## 方法

1. 子代理给出"严重度 + file:line + 具体失败场景"。
2. 主会话**读真实代码**确认结构性问题是否存在。
3. 对可观测的（如限流、认证），**线上实测复现**，而不是凭代码推断。
4. 区分"当前可利用" vs "结构性隐患/防回归"，分别定级。

## 误报 限流并非失效

> [!failure] 子代理结论（错误）
> "没有 `--proxy-headers`，所有请求都按 `127.0.0.1` 计数 → 共用一个桶 → 既能 DoS 锁死所有人，又能伪造 XFF 绕过。"

**实测推翻**：

- 连发 40 次坏登录 → `30×401 → 10×429`，且日志里限流 key 是**真实公网 IP**，不是 `127.0.0.1`。
- 40 次各带**不同伪造 `X-Forwarded-For`** → 仍按真实 IP 计数（伪造值被忽略）。

> [!important] 根因：uvicorn 默认开 `--proxy-headers`
> uvicorn **默认启用** proxy-headers 且默认信任 `127.0.0.1`，所以会读 nginx 的 `X-Forwarded-For` 把 `request.client.host` 设成真实客户端 IP。nginx 用 `$proxy_add_x_forwarded_for` **追加**真实 IP 作可信末跳，uvicorn 取它 → 伪造无效。子代理误以为"不加 flag 就拿不到真实 IP"，事实相反。

## 半误报：JWT 空密钥"严重"

子代理标 CRITICAL（token 伪造）。实测**生产 `.env` 已设置 JWT_SECRET**，不可利用 → 降级为"加启动守卫防回归"（见 [[安全加固清单#启动即校验关键密钥]]）。

## 一个排查小技巧

> [!tip] 限流没触发 ≠ 限流坏了
> 第一次测时发 35 次全 401、无 429，差点误判"限流彻底失效"。实为请求**跨了固定窗口的分钟边界**，每个窗口都没到阈值。一次性发到**明显超过阈值**（40 > 30）才稳定复现 429。测速率限制要用干净的单窗口爆发。

## 经验

> [!quote] 一句话
> 审查代理擅长**定位可疑点**，不擅长**断言后果**。可观测的结论一律线上复现；不可观测的（如竞态窗口）至少把代码路径读通、把缺失的守卫指实。
